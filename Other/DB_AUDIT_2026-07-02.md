## 🔴 HIGH — order marked "printed" but silently missing a matchable item

**Order `260702J4B2SETQ`** (Shopee, customer `whackowilson`, `2026-07-01`, `overall_order_status = printed`) has an `order_items` row:
- `variant_sku = BLO-SW-10225-WM`, `variant_id = NULL`, `variant_name = "item does not exist"`, `item_print_status = not_applicable`, never sent to print.

But **the catalog match exists**: product `BLO-SW-10225` ("R2-D2 - 10225") and variant `BLO-SW-10225-WM` ("R2-D2 - 10225 - WM") are both present and valid in the DB. The matcher failed to resolve a SKU that was actually resolvable, and because `item_print_status` landed on `not_applicable` it was silently excluded from the print queue rather than flagged as a failure. **The order is already marked `printed` at the order level, so this specific R2-D2 unit may never have been printed.**

**Action needed:** manually verify whether the R2-D2 WM unit for this order was printed; if not, create the order_item/print_job by hand and re-dispatch. Also worth checking Scout's matching logs for why an exact-SKU match failed here — this looks like a matching-pipeline bug, not a data problem, so it could recur on other orders.

## 🟡 MEDIUM — data-generation bug: wrong `set_number` on one product's variants

Product `BLO-SW-75301-DS-TALL` ("Luke Skywalkers X-Wing Fighter - 75301 - DS-TALL-1") has 6 variants (`BLO-SW-75301-DS-TALL-1` through `-6`). All 6 should share `set_number = 75301` (same physical LEGO set, just different plaque-count copies), but instead each was auto-incremented:

| variant_sku | set_number (found) | set_number (expected) |
|---|---|---|
| BLO-SW-75301-DS-TALL-1 | 75303 | 75301 |
| BLO-SW-75301-DS-TALL-2 | 75304 | 75301 |
| BLO-SW-75301-DS-TALL-3 | 75305 | 75301 |
| BLO-SW-75301-DS-TALL-4 | 75306 | 75301 |
| BLO-SW-75301-DS-TALL-5 | 75307 | 75301 |
| BLO-SW-75301-DS-TALL-6 | 75308 | 75301 |

This is the only product in the whole DB with this pattern (checked all 341 products — no other product has sibling variants disagreeing on `set_number`), so it looks like a one-off bug in whatever generated this product's plaque-count tree (`variant_name` is also identically "...DS-TALL-1 - DS" on all 6 rows instead of being indexed). `set_number` isn't used by the matching pipeline directly, so this likely hasn't caused mis-prints, but it will throw off any future set-number-based lookups or reporting for this product.

**Action needed:** update `set_number` to `'75301'` on all 6 rows. Say the word and I'll apply it.

## 🟢 LOW — worth a look, not urgent

### `print_files.simplyprint_file_id` shared across different products (3 groups)
Distinguished from the 100+ *expected* sharing groups below by requiring different `set_number` **and** a non-generic filename:

1. **`BLO-OTR-71047-DS`** (Dungeons & Dragons - 71047) and **`BLO-MFG-DS-CMF-12`** (Minifig - DS - CMF - 6) both point at `Minifig-CMF - DS-12 - 50g-84m-a1`. The D&D set using a generic "CMF" (Collectible Minifigures) branded stand file looks like a mismatch rather than intentional reuse — worth confirming the D&D variant has its own correct file.
2. **`BLO-SW-7669-DS-NP`** and **`BLO-SW-75214-DS-NP`** (two different Anakin's Jedi Starfighter sets, different physical scale) both point at a file explicitly named `...75214 - DS-NP...`. Same character, different LEGO set — the 7669 variant is likely using the wrong file (analogous to the C-2 category fixed in the prior audit).
3. **`BLO-SW-75301-DS-TALL-1..6`** all point at a file named `...75301 - DS-T...` — this one's fine, it's the *same* product (see bug above), just listed here because it tripped the "different set_number" heuristic due to the bug.

### Order items with `item_print_status = not_applicable` and no catalog match
3 more line items across 2 orders (`260701J14TN74Y`, `2606252FF623VE`) have `variant_sku = "item does not exist"` — these look like genuinely non-catalog items (e.g. bundled/free items Shopee includes that aren't physical SKUs), correctly excluded via `not_applicable`. No action needed, but flagging since it's the same shape as the HIGH finding above — worth spot-checking these aren't also silently-missed real items.

### 45 `print_files` rows with `weight_g = 0` and/or `print_time_m = 0` (excluding "to be made" placeholders)
Real, named files (mostly the `(2)` alternate-part files for UCS sets — Dune Ornithopter, NASA Artemis, NASA Shuttle Carrier, SC-VDS-F1 series, Venator) that never got weight/print-time metadata filled in. Cosmetic — doesn't block dispatch — but worth backfilling for accurate cost/time estimates.

### 9 variants have `plaque_count` set despite `variant_type = 'WM'`
Per the documented invariant (`plaque_count` should only be set for `DS-N` variants, `NULL` for `WM`/`DS-NP`/`FWM`/`BASE`), these 9 rows (e.g. `BLO-SW-75049-DS-1` through `-4`, `BLO-SW-75374-DS-1`, `BLO-SW-7915-DS-6`, `BLO-SW-75314-DS-2`, `BLO-SW-75414-DS-1`, `BLO-SW-7931-DS-1`) are mistyped — the SKU pattern (`-DS-N`) suggests these should be `variant_type = 'DS'`, not `'WM'`. Low impact since `variant_type` isn't the sole matching key, but worth a bulk correction.

### 5 of 12 orders have `shop_id = NULL` ("Unassigned")
Expected per the multi-shop Phase 0/1 design (Scout couldn't attribute to a shop), but 42% unassigned is a high proportion — worth checking Scout's shop-routing logic if this keeps climbing as order volume grows.

## ✅ Clean (no action needed)

- **Referential integrity: perfect.** Zero orphaned or NULL-where-required foreign keys across all 11 relational tables (`variants→products`, `print_files→variants`, `listings→products`, `listing_variations→listings/variants`, `orders→shops`, `products→shops`, `order_items→orders/variants`, `print_jobs→order_items/print_files`).
- **No duplicate `master_sku`, `variant_sku`, `platform_listing_name`, `platform_order_id`, `shops.slug`, or `shops.sku_prefix`.**
- **No `UNIQUE(listing_id, platform_variation_name)` violations.**
- **No required fields NULL/empty** across products, variants, listings, orders, print_files.
- **No status values outside their CHECK-constraint allowed sets** — `overall_order_status`, `waybill_processing_status`, `item_print_status`, `job_execution_status` are all clean (the v9 migration's schema-drift concern about missing constraints didn't manifest as bad data here).
- **No negative/zero prices, weights, print times, subtotals, or quantities.**
- **No `updated_at < created_at` anywhere.**
- **SKU-prefix consistency:** every product/variant SKU correctly starts with its shop's `sku_prefix` (`BLO-`).
- **Print-dispatch consistency:** no `print_jobs` with a non-pending status but missing `simplyprint_job_id`; no `print_jobs` pointing at a `print_files` row with a NULL `simplyprint_file_id`; no `completed` order_items missing a `print_jobs` row; no `pending` order_items whose parent order is already `completed`.
- **107 `simplyprint_file_id` collisions found, 104 of them by design:**
  - 97 groups (514 rows) — same `set_number`, i.e. the same physical LEGO set's different plaque-count/orientation variants correctly sharing one print file.
  - 6 groups (386 rows) — different `set_number`s but a generic/universal filename (the `N-Starwars - PLATE` inserts, common minifig bases, etc.) — these are genuinely shop-wide reusable accessory components, not mismatches.
  - Only the 3 groups flagged in 🟢 above fall outside both patterns.
