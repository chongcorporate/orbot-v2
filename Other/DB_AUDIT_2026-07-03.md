## 🔴 HIGH — live listings mapped to the wrong physical set

### H-1. "Wall Mount For Lego Star Wars R2-D2 (75308)" → prints the 10225 wall mount
The active listing (product `BLO-SW-75308`) has its variations (including one with an **empty variation name** `''`) mapped to **`BLO-SW-10225-WM`** — a different R2-D2 set. Both variants exist with distinct, real print files:
- `BLO-SW-75308-WM` → `R2-D2-75308 - WM - 120g-178m-a1` (unmapped in any listing!)
- `BLO-SW-10225-WM` → `R2-D2-10225 - WM - 120g-178m-a1` ← what this listing dispatches

Any order on this listing prints the **10225** geometry for a customer who owns **75308**. Almost certainly related to yesterday's HIGH finding (order `260702J4B2SETQ` carrying `variant_sku = BLO-SW-10225-WM`).

**Fix:** repoint both variation rows to `BLO-SW-75308-WM`; give the empty-string variation row a real name (or delete it).

### H-2. "Display Stand for Lego Star Wars Droid Tri-Fighter (8086)" → prints the 75044 stand
The active listing (product `BLO-SW-8086`) maps `Base - Plaque, 1–6` and `Base - Blank, 7 - Blank` to **`BLO-SW-75044-DS-*`** — the *2014* Droid Tri-Fighter, a different mould (`...75044 - DS - 51g-122m` vs `...8086 - DS - 55g-120m`, different SimplyPrint IDs). The correct `BLO-SW-8086-DS-1..6` / `-DS-NP` variants exist with their own real print files but are **unmapped**.

**Fix:** repoint the 7 variation rows from `75044` variants to the matching `8086` variants.

## 🟡 MEDIUM — mis-keyed records & cross-wired listings

### M-1. Duplicate product `BLO-SW-75218-DS` is really Vonreg's TIE 75240 — and is wired into the X-Wing listing
Product `BLO-SW-75218-DS` ("Major Vonreg's TIE Fighter - 75240 - DS-SHORT-1") holds `BLO-SW-75218-DS-1..6`, all with `set_number = 75240`. The SKU says 75218 (Luke's X-Wing); the content is Vonreg's TIE short stands. It duplicates the proper `BLO-SW-75240` product. Consequences, in two live listings:
- **Vonreg listing** `(75240)`: `Base-Plaque - Short, 1–6` → `BLO-SW-75218-DS-1..6` (right content, wrong SKU), while the correct `BLO-SW-75240-DS-SHORT-1..6` sit unmapped.
- **X-Wing listing** `(75301 / 75218)`: `Base - Plaque, 1–6` → the same `BLO-SW-75218-DS-1..6` — an X-Wing customer would get a **Vonreg TIE short stand**. Meanwhile `BLO-SW-75301-DS-1..6` sit unmapped.

**Mitigating factor:** all `BLO-SW-75218-DS-*` print_files are `"to be made"` placeholders with no SimplyPrint ID, so nothing wrong has printed — an order would stall at dispatch instead of mis-printing. Still, the wiring is wrong on both listings.

**Fix:** repoint Vonreg-listing Short rows → `BLO-SW-75240-DS-SHORT-1..6`; repoint X-Wing-listing plaque rows → the correct X-Wing variants (`BLO-SW-75301-DS-1..6` or a true 75218 set); then retire the `BLO-SW-75218-DS` product + its 6 variants.

### M-2. ISS master_sku digit-swap + duplicate variant
Product `BLO-IDS-21312` is named "ISS International Space Station - **21321**" (21321 is the real ISS set; 21312 is Women of NASA). It holds **two duplicate WM variants**: `BLO-IDS-21312-WM` and `BLO-IDS-21321-WM`, both `set_number = 21321`.
**Fix:** rename master_sku → `BLO-IDS-21321`, merge/delete one WM variant (check which one listings & print_files point at first).

### M-3. NASA Apollo LRV SKU says 42152, everything else says 42182
Product `BLO-TCH-42152` / variant `BLO-TCH-42152-WM` are named "NASA Apollo Lunar Roving Vehicle LRV - **42182**". The variant also has **zero print_files rows** (the only variant in the DB with none) and no listing.
**Fix:** rename both SKUs to `...42182...`, set `set_number = 42182`, add print file when ready.

### M-4. D&D 71047 theme-prefix three-way disagreement
Product `BLO-MFG-71047` ("Dungeons & Dragons", category Minifig) ↔ variant `BLO-OTR-71047-DS` ↔ `print_files.variant_sku = BLO-MFG-71047-DS` (2 rows, denorm column matches neither the variant's actual SKU). Any SKU-string-based file lookup misses. Same product yesterday flagged for pointing at the generic `Minifig-CMF - DS-12` file.
**Fix:** pick one theme code (MFG or OTR) and align product, variant, and both print_files rows.

### M-5. `variant_type` + `variant_name` scrambled on three products (16 rows)
Sibling variants have shuffled type/name values that contradict their own SKUs:
- `BLO-SW-75348-DS-S-1..4,-6`, `-DS-T-6`, `-DS-NP-S` typed `WM` (named "… - WM"); `-DS-S-5` typed `DS-NP`; `-DS-NP-T` typed `DS` — 9 rows (TIE Interceptor & Fang Fighter)
- `BLO-SW-75323-DS-TALL-1` typed `WM`, `-DS-TALL-2`/`-DS-SHORT-1` typed `DS-NP`, `-DS-NP-SHORT`/`-DS-NP-TALL` typed `DS` — 5 rows (The Justifier)
- `BLO-SW-9493-DS-NP-SHORT` typed `DS`, `-DS-SHORT-6` typed `WM` — 2 rows (X-Wing 9493)

Looks like a column-shift during that batch's ingest. Low matching impact (type isn't a match key) but poisons any type-based filtering/reporting.

### M-6. Leftover from yesterday's set_number increment bug
`BLO-SW-75301-DS-NP-TALL` still has `set_number = 75302`. Yesterday's fix corrected `-DS-TALL-1..6` but this sibling NP row was missed.
**Fix:** one-row update to `75301`.
